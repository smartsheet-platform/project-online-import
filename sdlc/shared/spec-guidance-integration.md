# Specification Guidance Integration

## Overview

Guidelines for using project-specific documentation templates and standards when creating specifications. Ensures consistent specification format and quality across the project.

## Guidance Directory

**Location**: `sdlc/docs/specs/` (including all subdirectories)

**Purpose**: Central location for specification templates, standards, examples, and style guides

## When to Reference Guidance

- Starting any specification or documentation task
- User requests adherence to documentation standards
- Creating specifications that should follow established templates
- Need clarification on documentation format or requirements

## How to Use Guidance

### Step 1: Check for Guidance Documents
Check for guidance documents in `sdlc/docs/specs/` including all subdirectories

### Step 2: Read Applicable Templates
Read applicable templates, standards, or examples from all subdirectories

### Step 3: Apply Guidance
Apply guidance to current specification work

### Step 4: Create Specifications
CREATE new specifications in `sdlc/docs/specs/` following established patterns

### Step 5: Organization
Use subdirectories to organize different types of specifications

## Guidance File Types

### Specification Template
- **Filename**: `specification-template.md`
- **Usage**: Reference for standard specification structure and format
- **When**: Creating any new specification document

### Documentation Standards
- **Filename**: `documentation-standards.md`
- **Usage**: Reference for style guide, terminology, and quality requirements
- **When**: Writing any documentation or specification content

### API Spec Template
- **Filename**: `api-spec-template.md`
- **Usage**: Reference for API integration specification format
- **When**: Documenting API integrations or external service dependencies

### Specification Examples
- **Filename**: `good-specification-examples.md`
- **Usage**: Reference examples of well-written specifications from your project
- **When**: Need guidance on specification quality or completeness

### Review Checklist
- **Filename**: `review-checklist.md`
- **Usage**: Reference for specification validation and quality assurance
- **When**: Completing specifications or preparing for review

### Domain Vocabulary
- **Filename**: `domain-vocabulary.md`
- **Usage**: Reference for project-specific terminology and language
- **When**: Writing specifications that need consistent terminology

## Integration Workflow

### Startup Check
1. Check if `sdlc/docs/specs/` directory exists
2. List available guidance documents and templates
3. Read relevant templates based on specification type
4. Apply templates and standards to specification work

### Specification Guidance
1. Use specification templates for consistent structure
2. Follow documentation standards for style and format
3. Apply domain vocabulary for consistent terminology
4. Use examples as quality benchmarks
5. Apply review checklist before completion

## Quality Assurance

### Template Compliance
- [ ] Specification follows template structure from guidance documents
- [ ] Documentation meets standards defined in guidance
- [ ] Terminology matches domain vocabulary
- [ ] Quality meets standards shown in examples

### Consistency Validation
**Principle**: All specifications should follow the same patterns and standards

**Implementation**: Reference guidance documents to maintain consistency across all specs

## User Communication

### When Guidance Available
```
I found project-specific specification guidance in sdlc/docs/specs/. I'll use these
templates and standards to create documentation that matches your project's
format and quality requirements.
```

### When No Guidance
```
No project-specific specification guidance found in sdlc/docs/specs/. I'll create
documentation using general best practices. Consider adding specification-template.md
and documentation-standards.md for more consistent results.
```

## Best Practices

1. **Always check for and apply project-specific specification guidance**
2. **Maintain consistency with established documentation patterns**
3. **Suggest improvements to guidance when patterns need clarification**
